## 1. Shared utilities (build first)

### 1.1 `utils/DiscordWebhook.java`
Pure Java, fully async, no extra dependencies. Every notifying module gets a `webhook-url` StringSetting (default empty). If blank, the helper no-ops. A global default lives in `NecronConfig`.

```java
package com.example.addon.utils;
import java.net.URI;
import java.net.http.*;
import java.time.Duration;
import java.util.concurrent.CompletableFuture;

public final class DiscordWebhook {
    private static final HttpClient CLIENT =
        HttpClient.newBuilder().connectTimeout(Duration.ofSeconds(10)).build();
    private DiscordWebhook() {}

    private static String escape(String s) {
        if (s == null) return "";
        StringBuilder b = new StringBuilder();
        for (char c : s.toCharArray()) switch (c) {
            case '"'  -> b.append("\\\"");
            case '\\' -> b.append("\\\\");
            case '\n' -> b.append("\\n");
            case '\r' -> b.append("\\r");
            case '\t' -> b.append("\\t");
            default   -> { if (c < 0x20) b.append(String.format("\\u%04x",(int)c)); else b.append(c); }
        }
        return b.toString();
    }

    public static void sendMessage(String url, String content) {
        if (url == null || url.isBlank()) return;
        post(url, "{\"content\":\"" + escape(content) + "\"}");
    }

    public static void sendEmbed(String url, String title, String description, int color) {
        if (url == null || url.isBlank()) return;
        post(url, "{\"embeds\":[{\"title\":\"" + escape(title)
            + "\",\"description\":\"" + escape(description)
            + "\",\"color\":" + (color & 0xFFFFFF) + "}]}");
    }

    private static void post(String url, String json) {
        HttpRequest req = HttpRequest.newBuilder().uri(URI.create(url))
            .header("Content-Type","application/json").timeout(Duration.ofSeconds(10))
            .POST(HttpRequest.BodyPublishers.ofString(json)).build();
        CompletableFuture.runAsync(() -> {
            try { CLIENT.send(req, HttpResponse.BodyHandlers.discarding()); }
            catch (Exception ignored) {}
        });
    }
}
```

### 1.2 `utils/XaeroWaypointUtil.java`
Appends to Xaero's on-disk waypoint file. Only acts if Xaero Minimap is loaded. Format is version-sensitive — if waypoints don't appear, user should create one manually and compare the file format.

```java
package com.example.addon.utils;
import net.fabricmc.loader.api.FabricLoader;
import net.minecraft.client.Minecraft;
import net.minecraft.client.multiplayer.ServerData;
import java.nio.file.*;

public final class XaeroWaypointUtil {
    private XaeroWaypointUtil() {}

    public static boolean isXaeroLoaded() {
        var fl = FabricLoader.getInstance();
        return fl.isModLoaded("xaerominimap") || fl.isModLoaded("xaerominimapfair");
    }

    /** colorIndex 0-15. dimension: 0=OW, -1=Nether, 1=End. */
    public static void addWaypoint(String name, int x, int y, int z, int colorIndex, int dimension) {
        if (!isXaeroLoaded()) return;
        try {
            Path dir = resolveDir(dimension);
            Files.createDirectories(dir);
            Path file = dir.resolve("waypoints.txt");
            String initials = name.isEmpty() ? "X" : name.substring(0,1).toUpperCase();
            String line = "waypoint:" + name + ":" + initials + ":" + x + ":" + y + ":" + z
                + ":" + (colorIndex & 15) + ":false:0:gui.xaero_default:false:0:false"
                + System.lineSeparator();
            Files.writeString(file, line, StandardOpenOption.CREATE, StandardOpenOption.APPEND);
        } catch (Exception ignored) {}
    }

    private static Path resolveDir(int dimension) {
        Minecraft mc = Minecraft.getInstance();
        ServerData sd = mc.getCurrentServer();
        String server = (sd != null) ? "Multiplayer_" + sd.ip.replace(':', '_') : "Singleplayer";
        String dim = switch (dimension) { case -1 -> "dim%-1"; case 1 -> "dim%1"; default -> "dim%0"; };
        return FabricLoader.getInstance().getGameDir()
            .resolve("XaeroWaypoints").resolve(server).resolve(dim);
    }
}
```

### 1.3 `utils/PacketLimiter.java`
Global per-tick cap on self-generated packets. Call `PacketLimiter.resetTick()` once per tick from a single always-on TickEvent handler in AddonTemplate. Use `PacketLimiter.send(packet)` for all extra packets modules generate (not the vanilla move packet).

```java
package com.example.addon.utils;
import net.minecraft.network.protocol.Packet;
import static meteordevelopment.meteorclient.MeteorClient.mc;

public final class PacketLimiter {
    private static int sentThisTick = 0;
    private static int maxPerTick   = 8;
    private PacketLimiter() {}

    public static void resetTick()          { sentThisTick = 0; }
    public static void setMaxPerTick(int n) { maxPerTick = Math.max(1, n); }

    /** Sends the packet if under budget. Returns false and drops if over. */
    public static boolean send(Packet<?> packet) {
        if (mc.player == null || mc.player.connection == null) return false;
        if (sentThisTick >= maxPerTick) return false;
        mc.player.connection.send(packet);
        sentThisTick++;
        return true;
    }
}
```

## 2. Modules

### 2.1 StashFinder
**Class:** `modules/basefinding/StashFinder.java`
**Description:** Scans loading chunks for clusters of storage blocks and reports potential stashes.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `storage-blocks` | BlockListSetting | Chest, TrappedChest, Barrel, all 16 ShulkerBox colors, EnderChest, Furnace, Hopper, Dispenser, Dropper | Block types that count toward a stash. |
| `min-cluster-size` | IntSetting 1–100 | 8 | Minimum storage blocks near each other before reporting. Filters lone chests. |
| `cluster-range` | IntSetting 1–16 | 4 | Blocks within this distance count as the same cluster. |
| `notify-chat` | BoolSetting | true | Print found stash coords to chat. |
| `add-waypoint` | BoolSetting | true | Add an Xaero waypoint at the stash centre (requires Xaero Minimap). |
| `send-webhook` | BoolSetting | false | Send a Discord embed when a stash is found. |
| `webhook-url` | StringSetting | "" | Discord webhook URL. Visible only when send-webhook is true. Falls back to NecronConfig if blank. |
| `save-to-file` | BoolSetting | true | Append finds to necron/stashes.csv in the game directory. |

**Logic:**
1. On `ChunkDataEvent`, queue the chunk position (do not scan inline — never block ChunkDataEvent).
2. In `TickEvent.Post`, pop 1–2 queued chunks and scan block entities via `chunk.getBlockEntities()`. Count entries whose block is in `storage-blocks`.
3. Group positions into clusters: bucket by `(x / cluster-range, z / cluster-range)`, merge adjacent buckets with ≥1 block each. If cluster size ≥ `min-cluster-size`, it is a stash.
4. Track reported clusters in a `Set<BlockPos>` (keyed by rounded centre) per session to avoid duplicates.
5. On a new stash: chat if enabled; `XaeroWaypointUtil.addWaypoint("Stash", cx, cy, cz, 12, dim)`; `DiscordWebhook.sendEmbed(url, "Stash Found", coords+blockCount, 0x00FF00)`; append CSV line `timestamp,dimension,x,y,z,blockCount` to `necron/stashes.csv`.

Derive the dimension int: `mc.level.dimension() == Level.NETHER` → -1; `Level.END` → 1; else 0.

**Packet-kick prevention:** Sends zero packets — read-only. Only risk is main-thread lag from scanning too many chunks per tick. Cap at 1–2 chunks per tick.

**README tutorial:**
> **StashFinder** scans chunks as they load for clusters of storage blocks. When a group exceeds *Min Cluster Size*, it reports the location.
> - *Storage Blocks* — which block types count.
> - *Min Cluster Size* — how many must be near each other. Higher = fewer false positives.
> - *Cluster Range* — how close blocks must be to count as one stash.
> - *Add Waypoint* — drops an Xaero waypoint (needs Xaero's Minimap installed).
> - *Send Webhook + Webhook URL* — Discord ping on each find (see Discord Webhooks section).
> - *Save To File* — logs to `.minecraft/necron/stashes.csv`.
> **Getting kicked?** Not possible — read-only.

### 2.2 NewChunkDetector
**Class:** `modules/basefinding/NewChunkDetector.java`
**Description:** Highlights chunks generated by newer world generation. Fresh chunks mark recent player travel paths.

**Detection rules (no extra packets):**
- **Overworld:** old chunks (pre-1.18) have bedrock at y≈0 with void below; new chunks (1.18+) have deepslate below y=0. Check: if any block at y < 0 in the chunk is deepslate → NEW.
- **Nether:** old (pre-1.16) lack basalt/blackstone/soul_soil/ancient_debris. Presence of any → NEW.
- Auto mode picks the rule by current dimension.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `mode` | EnumSetting {Auto, Overworld, Nether} | Auto | Detection ruleset. Auto picks by dimension. |
| `render` | BoolSetting | true | Draw a coloured overlay on new chunks. |
| `new-chunk-color` | ColorSetting | green low alpha | Colour for new-chunk overlay. |
| `shape-mode` | EnumSetting ShapeMode | Both | Lines / Sides / Both. |
| `render-height` | IntSetting 1–384 | 256 | Vertical height of the overlay box. |

**Logic:**
1. On `ChunkDataEvent`, run the signature check (sample one column; bail as soon as a deciding block is found).
2. Store results in `Map<ChunkPos, Boolean>` (isNew). Evict chunks beyond render distance to bound memory.
3. On `Render3DEvent`, for each NEW chunk near the player draw a 16×16×`render-height` box with `new-chunk-color`.

Expose `public static Map<ChunkPos, Boolean> getChunkMap()` so ChunkTrailFollower can read it.

**Packet-kick prevention:** Read-only — zero risk.

**README tutorial:**
> **NewChunkDetector** highlights chunks from newer terrain generation. Players generate new chunks as they travel — follow them toward bases using ChunkTrailFollower.
> - *Mode* — Auto picks by dimension. Overworld uses deepslate-below-y0; Nether uses 1.16-block presence.
> - *Render / Color / Shape / Height* — boundary overlay controls.
> **Getting kicked?** Not possible — read-only.

### 2.3 ChunkTrailFollower
**Class:** `modules/basefinding/ChunkTrailFollower.java`
**Description:** Follows the edge of newly generated chunks automatically. Requires NewChunkDetector to be active.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `scan-radius` | IntSetting 2–12 | 6 | Chunk radius to search for the new-chunk frontier each step. |
| `lost-trail-action` | EnumSetting {Stop, Notify, Disconnect} | Notify | What to do when the trail ends (you have likely arrived). |
| `webhook-url` | StringSetting | "" | Discord webhook for trail-lost events. |

**Logic:**
1. Each tick, read `NewChunkDetector.getChunkMap()`. Among new chunks within `scan-radius`, pick the one furthest along the current travel heading (maintain a heading Vec3 from recent position delta).
2. Issue a Baritone `#goto <x> <z>` command via the chat command bus (avoids a hard Baritone API dependency). Gate this behind a check that Baritone is installed (`FabricLoader.getInstance().isModLoaded("baritone")`); if not, only rotate the player toward the target.
3. When no new chunks remain ahead, fire `lost-trail-action` and send webhook.

**Packet-kick prevention:** Movement is vanilla/Baritone packets. Risk is the 15-min AFK kick on 2b2t if only walking — recommend pairing with an AntiAFK module.

**README tutorial:**
> **ChunkTrailFollower** chases the edge of newly generated chunks. Enable NewChunkDetector first, fly to where old meets new, then enable this.
> - *Scan Radius* — how far ahead to look for fresh chunks.
> - *Lost Trail Action* — Stop / Notify / Disconnect when the trail ends (likely arrived at a base).
> Requires Baritone for automatic pathfinding; without it, only rotates you toward the trail.
> **Getting kicked?** Long walks with no block interaction can hit 2b2t's 15-min AFK kick.

### 2.4 AutoElytraScanner
**Class:** `modules/basefinding/AutoElytraScanner.java`
**Description:** Flies a grid, spiral, or zigzag pattern over a defined area so StashFinder can sweep it hands-free.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `pattern` | EnumSetting {Grid, Spiral, ZigZag} | Grid | Flight path pattern. |
| `min-x` | IntSetting | 0 | West bound of scan area. |
| `max-x` | IntSetting | 0 | East bound. |
| `min-z` | IntSetting | 0 | North bound. |
| `max-z` | IntSetting | 0 | South bound. |
| `altitude` | IntSetting 0–320 | 250 | Flight height. |
| `leg-spacing` | IntSetting 16–512 | 128 | Distance between parallel passes (set near render distance in blocks). |
| `firework-delay` | IntSetting ticks 8–60 | 12 | Minimum ticks between firework uses. Do not set below 8. |
| `target-speed` | DoubleSetting | 30.0 | Desired horizontal speed bps; only boost when below this. |
| `pause-on-stash` | BoolSetting | true | Pause when StashFinder reports a find. |
| `webhook-url` | StringSetting | "" | Optional status updates (start, stuck, done). |

**Logic:**
1. On activate, compute waypoints for the chosen pattern across the bounds at `altitude`, spaced by `leg-spacing`.
2. Each tick: ensure elytra is deployed (edge-trigger `START_FALL_FLYING` only when `!isFallFlying()`); steer toward the next waypoint by setting player rotation and holding forward.
3. Measure horizontal speed from position delta. If below `target-speed` AND ≥ `firework-delay` ticks since last firework: find a firework rocket in hotbar via `InvUtils.findInHotbar`, swap to it with `ServerboundSetCarriedItemPacket`, right-click (use item), restore held slot. Reset firework timer.
4. Advance waypoint when within `leg-spacing/4` blocks. On last waypoint, deactivate and send webhook "done".

**Packet-kick prevention:** Firework use-item packets are the risk. Never fire more often than `firework-delay` ticks (min 8). Use one firework per cycle. Swap via held-slot change (not shift-click). Enable PacketAntiKick.

**README tutorial:**
> **AutoElytraScanner** flies a pattern over an area so StashFinder can sweep it automatically.
> - *Pattern* — Grid (lawnmower), Spiral, or ZigZag.
> - *Min/Max X/Z* — the rectangle to cover.
> - *Altitude* — flight height (use ~250 on Nether roof).
> - *Leg Spacing* — gap between passes; match to your render distance.
> - *Firework Delay* — ticks between boosts. **Keep ≥ 8 or 2b2t kicks you.**
> - *Target Speed* — only boosts when slower than this.
> - *Pause On Stash* — stops when StashFinder finds something.
> **Getting kicked?** From fireworks firing too fast. Keep Firework Delay ≥ 8 and run PacketAntiKick.

### 2.5 PortalFinder
**Class:** `modules/basefinding/PortalFinder.java`
**Description:** Detects destroyed/used nether portals and player digging in the Nether by finding pockets of regular air (Blocks.AIR) inside naturally cave-air terrain.

**Key distinction:** use `Blocks.AIR` exact match, NOT `state.isAir()` — `isAir()` also matches cave_air and void_air. A cluster of regular `Blocks.AIR` surrounded by `Blocks.CAVE_AIR` = player activity.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `min-air-cluster` | IntSetting 4–200 | 12 | Minimum regular-air blocks in a cluster before reporting. |
| `render` | BoolSetting | true | Draw a box around detected clusters. |
| `color` | ColorSetting | purple mid-alpha | Overlay colour. |
| `add-waypoint` | BoolSetting | true | Xaero waypoint on detection. |
| `webhook-url` | StringSetting | "" | Discord notification. |

**Logic:**
1. Only run in the Nether (`mc.level.dimension() == Level.NETHER`).
2. On `ChunkDataEvent`, scan block states in the chunk for `state.getBlock() == Blocks.AIR` (exact). Collect those positions.
3. Flood-fill adjacent regular-air blocks into clusters. If cluster size ≥ `min-air-cluster`, report it.
4. Render box, Xaero waypoint, webhook, deduplicate per session.

**Packet-kick prevention:** Read-only — zero risk.

**README tutorial:**
> **PortalFinder** spots nether portals and player digging in the Nether by finding pockets of regular air inside natural cave-air. Nether-only.
> - *Min Air Cluster* — how large the pocket must be (raise to ignore small tunnels).
> - *Render / Color* — the overlay.
> - *Add Waypoint / Webhook* — same as StashFinder.
> **Getting kicked?** No — read-only.

### 2.6 VelocityBoost
**Class:** `modules/movement/VelocityBoost.java`
**Description:** Keybind-triggered speed burst in the look direction. Spread over multiple ticks to stay within 2b2t's movement tolerance.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `keybind` | KeybindSetting | none | Press to trigger a boost. |
| `strength` | DoubleSetting 0.1–4.0 | 1.5 | Boost magnitude in look direction. |
| `vertical` | DoubleSetting -1.0–2.0 | 0.2 | Extra upward velocity. |
| `ticks` | IntSetting 1–10 | 3 | Spread boost over this many ticks. Higher = safer. |
| `require-elytra` | BoolSetting | true | Only boost while gliding. |

**Logic:**
On keybind press start a counter = `ticks`. While counter > 0, each `TickEvent.Pre`: add velocity. Compute look vector from player yaw/pitch; call `mc.player.setDeltaMovement(vx + lookX * strength/ticks, vy + vertical/ticks, vz + lookZ * strength/ticks)`. Decrement counter. Let vanilla send the resulting movement packet — do not craft raw position packets.

**Packet-kick prevention:** Spreading over `ticks` (default 3) keeps per-tick delta small. Do not set strength very high with ticks=1.

**README tutorial:**
> **VelocityBoost** gives a burst of speed in your look direction — useful escaping fights or crossing gaps.
> - *Keybind* — trigger key.
> - *Strength* — boost power.
> - *Vertical* — extra lift.
> - *Ticks* — spread duration. **Higher is safer against speed kicks.**
> - *Require Elytra* — only while gliding.
> **Getting kicked?** Only if Strength is high with Ticks=1. Keep Ticks ≥ 3.

### 2.7 AutoTrap
**Class:** `modules/combat/AutoTrap.java`
**Description:** Places obsidian or ender chests around the nearest player in range. Highest packet-rate module — rate-limit aggressively.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `target-range` | DoubleSetting 1–6 | 4.5 | Max distance to target player. |
| `block` | EnumSetting {Obsidian, EnderChest, Auto} | Obsidian | Block to use. Auto tries obsidian then ender chest. |
| `top-only` | BoolSetting | false | Only place the two blocks above the head (faster, less material). |
| `blocks-per-tick` | IntSetting 1–4 | 1 | Placements per tick. Keep at 1 on 2b2t. |
| `rotate` | BoolSetting | true | Server-side rotate to each placement. |
| `reach` | DoubleSetting 3–5 | 4.5 | Max reach for block placement. |

**Logic:**
1. Each tick, find nearest other player within `target-range`. If none, idle.
2. Compute trap positions: four sides + top of target's head block (or just two top blocks if `top-only`). Filter to air/replaceable blocks within `reach`.
3. Find the trap block in hotbar (`InvUtils.findInHotbar`). Place up to `blocks-per-tick` positions using Meteor's `BlockUtils.place` with rotate flag.

**Packet-kick prevention:** Keep `blocks-per-tick` = 1. Never shift-click to refill mid-combat. Run PacketAntiKick. Confirm BlockUtils.place signature in Meteor jar before using.

**README tutorial:**
> **AutoTrap** boxes the nearest player in obsidian.
> - *Target Range* — how close they must be.
> - *Block* — Obsidian, Ender Chest, or Auto.
> - *Top Only* — just cap the head (faster).
> - *Blocks Per Tick* — **keep at 1 on 2b2t.** Higher = kick.
> - *Rotate* — faces each block placement.
> **Getting kicked?** Yes, easily if Blocks Per Tick > 1. Run PacketAntiKick alongside.

### 2.8 PacketAntiKick
**Class:** `modules/player/PacketAntiKick.java`
**Description:** Cancels invalid inventory click packets before they leave the client, preventing the most common 2b2t inventory-kick.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `block-invalid-quickmove` | BoolSetting | true | Cancel QUICK_MOVE clicks on empty or out-of-range slots. |
| `block-no-screen-clicks` | BoolSetting | true | Cancel inventory clicks when no container screen is open. |
| `max-clicks-per-tick` | IntSetting 1–20 | 8 | Hard cap on click packets per tick. |
| `log` | BoolSetting | false | Print to chat what was cancelled (for debugging). |

**Logic — on `PacketEvent.Send`:**
Use `ServerboundContainerClickPacket` and `ClickType` (Mojmap — see SPEC.md A.3 for exact code).
1. If `block-invalid-quickmove` and click type == QUICK_MOVE: get `mc.player.containerMenu`; if slot out of range or slot item is empty → cancel.
2. If `block-no-screen-clicks` and `!(mc.screen instanceof AbstractContainerScreen)` → cancel.
3. Maintain per-tick counter (reset in `TickEvent.Pre`); if counter > `max-clicks-per-tick` → cancel.
4. If `log`, print which rule fired.

**Packet-kick prevention:** This module IS the prevention. Keep enabled whenever running ElytraReplace, AutoElytraRestock, LoadoutSave, or AutoTrap.

**README tutorial:**
> **PacketAntiKick** prevents the most common 2b2t kick: invalid shift-click (QUICK_MOVE) packets. Automation modules sometimes click a slot a tick after it emptied — 2b2t kicks after a few of these. This cancels them before they send.
> - *Block Invalid QuickMove* — the main protection.
> - *Block No-Screen Clicks* — cancels clicks when no container is open.
> - *Max Clicks Per Tick* — hard ceiling.
> - *Log* — debug output.
> **Keep this on whenever using any inventory automation.**

### 2.9 ElytraReplace
**Class:** `modules/player/ElytraReplace.java`
**Description:** Automatically swaps in a fresh elytra when current durability drops to a threshold.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `durability-threshold` | IntSetting 1–432 | 20 | Swap when remaining durability hits this. |
| `source` | EnumSetting {Inventory, ShulkerInHotbar} | Inventory | Where to pull the spare elytra from. |
| `equip-chestplate-on-empty` | BoolSetting | false | Equip a chestplate if no spare elytra is found. |
| `click-delay` | IntSetting ticks 1–10 | 2 | Minimum ticks between inventory clicks during swap. |
| `chat-feedback` | BoolSetting | true | Notify when a swap occurs. |

**Logic:**
Use a state machine: IDLE → SWAPPING (one inventory click per `click-delay` ticks — pickup spare → place in armour → stow old). Re-validate source slot immediately before each click. **Packet-kick prevention:** One click per `click-delay` ticks. Re-validate before each click. Run PacketAntiKick.

**README tutorial:**
> **ElytraReplace** swaps in a fresh elytra before the current one breaks.
> - *Durability Threshold* — swap trigger point.
> - *Source* — pull from inventory or a shulker in hotbar.
> - *Equip Chestplate On Empty* — fallback if out of elytras.
> - *Click Delay* — ticks between clicks. **Higher is safer.**
> **Getting kicked?** Keep Click Delay ≥ 2 and run PacketAntiKick.

### 2.10 AutoElytraRestock
**Class:** `modules/player/AutoElytraRestock.java`
**Description:** Lands, builds a platform, opens ender chest, pulls fresh elytra + rockets, re-equips, and relaunches. Highest inventory-packet rate — use conservative delays.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `trigger-elytra-durability` | IntSetting 1–432 | 30 | Trigger when elytra durability drops to this. |
| `trigger-rockets` | IntSetting 0–64 | 4 | Trigger when firework count drops to this. |
| `platform-block` | EnumSetting {Obsidian, Netherrack, Cobblestone} | Obsidian | Block to stand on. |
| `step-delay` | IntSetting ticks 2–20 | 5 | Ticks between each automation step. Keep ≥ 5. |
| `clicks-per-step` | IntSetting 1–4 | 1 | Inventory clicks per step. Keep at 1 on 2b2t. |
| `webhook-url` | StringSetting | "" | Status notifications. |

**State machine steps (one transition per `step-delay` ticks):**
1. TRIGGER — conditions met
2. DESCEND — stop gliding, descend
3. PLACE_PLATFORM — place one platform block below
4. OPEN_CHEST — place + open ender chest
5. PULL_ITEMS — shift-click elytra then rockets (`clicks-per-step` per step, re-validate slot before each)
6. EQUIP — move fresh elytra to chest armour slot
7. RELAUNCH — close screen, jump, deploy elytra

Warn in chat if PacketAntiKick is not active when this module activates.

**Packet-kick prevention:** One click per step per `step-delay` ticks. Re-validate every slot. Keep `clicks-per-step` = 1. This is the easiest module to get kicked with — conservative defaults are mandatory.

**README tutorial:**
> **AutoElytraRestock** lands, restocks from your ender chest, and relaunches automatically when running low on elytra or rockets.
> - *Trigger Elytra Durability / Trigger Rockets* — when to restock.
> - *Platform Block* — what to stand on.
> - *Step Delay* — ticks between each action. **Keep ≥ 5.**
> - *Clicks Per Step* — **keep at 1 on 2b2t.**
> **Getting kicked?** This is the riskiest module. Max delays + PacketAntiKick always enabled.

### 2.11 ContainerPreview
**Class:** `modules/render/ContainerPreview.java`
**Description:** Shows the contents of a shulker box or container item by reading its NBT when hovered in inventory. Read-only — no packets.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `preview-shulkers` | BoolSetting | true | Preview shulker box contents on hover. |
| `preview-other` | BoolSetting | true | Preview other container items (chests held as items etc). |
| `show-counts` | BoolSetting | true | Show stack counts. |
| `scale` | DoubleSetting 0.5–2.0 | 1.0 | Popup size. |

### 2.12 LoadoutSave
**Class:** `modules/player/LoadoutSave.java`
**Description:** Save and restore inventory layouts. Commands: `.loadout save <name>`, `.loadout load <name>`, `.loadout list`, `.loadout delete <name>`.

**Settings (group: General)**
| Name | Type | Default | Description |
|---|---|---|---|
| `load-click-delay` | IntSetting ticks 1–10 | 3 | Ticks between clicks while restoring. |
| `match-by` | EnumSetting {ItemOnly, ItemAndNbt} | ItemOnly | Match by item type only, or exact NBT. |
| `chat-feedback` | BoolSetting | true | Notify on save/load. |

**Logic:**
- **Save:** record item type (+ NBT if `match-by=ItemAndNbt`) per slot 0–40. Write to `necron/loadouts/<name>.json`.
- **Load:** for each saved slot, if current slot matches skip. Otherwise find matching item elsewhere in inventory and move it — one click per `load-click-delay` ticks via a queued state machine. Re-validate before each click.

Register a Meteor command `LoadoutCommand` in `Commands.get().add(...)`.

**Packet-kick prevention:** Loading emits inventory clicks — one per `load-click-delay` ticks. Re-validate slots. Run PacketAntiKick.

**README tutorial:**
> **LoadoutSave** saves and restores inventory layouts.
> - Commands: `.loadout save <name>`, `.loadout load <name>`, `.loadout list`, `.loadout delete <name>`
> - *Load Click Delay* — ticks between rearranging clicks. **Higher is safer.**
> - *Match By* — Item Only (any sword fits) or Item And NBT (exact enchanted item).
> **Getting kicked?** Keep Load Click Delay ≥ 3 and run PacketAntiKick. Saving never sends packets.

### 3.1 ElytraDurabilityHud
**Class:** `hud/ElytraDurabilityHud.java`
**Description:** Shows current elytra durability on screen, coloured by threshold.

**Packet-kick prevention:** Read-only — zero risk.
