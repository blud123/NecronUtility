## 2. Fix + optimise `FastBreak`

Keep the packet-control + silent-tool design. Make these specific changes:

### 2a. Replace the `double-break` boolean with a 1–2 slider
2b2t's server (`ServerPlayerGameMode`) tracks **one active destroy + one delayed-destroy slot**, which is why at most 2 blocks can be packet-mined simultaneously and 3+ get rejected. Expose this as:
```java
IntSetting max-blocks, defaultValue 2, min 1, max 2, sliderRange(1,2)
```
`1` = primary only. `2` = allow the primary→secondary demotion (the STOP-hands-off-to-delayed-slot path you already have). Remove the old boolean and gate the secondary on `maxBlocks.get() >= 2`. (Verify the demotion still completes against `ServerPlayerGameMode` — the delayed slot is genuine, but confirm the STOP/START ordering.)

### 2b. Fix "only works while holding the right tool" — sync the tool to the server for the WHOLE break
Root cause: `applySilentSwap` only sends `ServerboundSetCarriedItemPacket` **near the finish** (`serverSwapTicks` window). But the server recomputes `getDestroyProgress` every server tick using whatever slot it currently believes you hold. If you're holding a non-tool for most of the break, the server's accumulated progress never reaches its ~0.7 completion threshold, so the finishing STOP is rejected → the block doesn't break unless you manually hold the tool.

Fix:
- When `swap-tool` is on, send `SetCarriedItem(bestSlot)` **at the moment the break starts** (first tick for that target), not near the finish. Hold that server slot for the entire break.
- On finish or abort, send `SetCarriedItem(originalSlot)` to restore.
- Keep the client-side `silentSlot` mixin so the local hotbar selection and break-progress maths still use the tool without the user's visible hotbar slot jumping. The server packet is what makes the break actually register; the mixin is just local cosmetics. (Other players will see the tool — that's unavoidable and fine.)
- Track `lastServerSlot` so you only send `SetCarriedItem` when it actually changes (you already do this — keep it). Delete the `serverSwapTicks` near-finish logic, or repurpose it to 0.

### 2c. Fix the expanding-cube render (it doesn't grow correctly)
Current render quantises to `breakingTicks` and recomputes `delta` per frame, so the cube jumps in steps and, when the tool wasn't synced (see 2b), finishes at the wrong time. Make it **time-based and smooth**:
- When a target starts breaking, record `long startMs = System.currentTimeMillis()` and `int estTicks = max(1, ceil(1f / breakDeltaForSlot(state, pos, bestSlot)))` on the target.
- In render: `float progress = clamp((System.currentTimeMillis() - startMs) / (estTicks * 50.0f), 0f, 1f);` (50 ms per tick). `half = progress * 0.5`. This grows continuously from a point at the block centre to the full 1×1×1 box, independent of tick boundaries.
- For instant breaks (`estTicks <= 1`), just render the full box for its single frame.
- Recompute `estTicks` only if the tool/state changes, otherwise lock it at start.

## 3. Fix + optimise `Nuker`

### 3a. Cap simultaneous breaks at 1–2
Replace `blocks-per-tick` (currently 1–4) with the same 1–2 concept as FastBreak:
```java
IntSetting max-blocks, defaultValue 2, min 1, max 2, sliderRange(1,2)
```
This is the number of **simultaneous in-progress** breaks, not per-tick throughput. Use it as the budget in every path. When delegating to FastBreak, cap at `min(maxBlocks.get(), 2)`.

### 3b. Make targets STICKY — lock a block until it's mined or leaves the radius
Right now Nuker re-picks the budget from the freshly sorted target list every tick, so a newly-appeared closer block can bump an in-progress block out of the budget and the cube/break "keeps changing." Fix the selection so in-progress breaks always keep their slot:

1. Each tick, build `targetSet` (all valid blocks in radius) as now.
2. Evict any locked/started block that is no longer in `targetSet` or is now air (free its slot, send ABORT if mid-break).
3. **Fill the budget in this order:** first, all currently-locked/started blocks that are still valid (these never get preempted); then, only if free slots remain, add new candidates from the sorted list.
4. Never re-sort or re-pick over a block that is mid-break. A locked block is released only when it finishes, leaves the radius, or becomes air.

`startedBreaks` already persists — extend it to be the single source of truth for "locked" slots and select around it as above. This also keeps FastBreak happy in the delegated path, because Nuker will feed it the same positions every tick instead of churning primaries.

### 3c. Same tool-sync fix as FastBreak (so it works without holding the tool)
In the standalone path, ensure the server-held slot equals the mining tool for the **whole** multi-tick break (set at START, restore when no breaks remain). You partly do this with `savedSlot`, but the logic has several restore points and churns. Simplify to one held-slot tracker: set it when the first locked break starts, restore it once `startedBreaks` is empty. In the FastBreak-delegated path, let FastBreak own the tool (don't double-swap).

### 3d. Performance — precompute the radius offset sphere
The biggest cost is the `(2r+1)³` block-state sweep + sort every tick. Precompute, **once whenever the radius setting changes**, an `int[][]` (or `BlockPos[]`) of offsets within the radius, pre-sorted for `CLOSEST` (and store the raw set for `LOWEST`/`HIGHEST` to sort cheaply). Each tick just offset from the player origin and read block states — no per-block `ceil`/`sqrt`/list allocation. Recompute only on radius change.

## 4. Fix `PacketAntiKick` (breaks when using shulkers)

Root cause: the `block-invalid-quickmove` guard checks `menu.getSlot(slot).getItem().isEmpty()`. But on `PacketEvent.Send` the client has **already applied the predicted shift-click locally**, so a legitimate shulker shift-click looks like a QUICK_MOVE on a now-empty slot and gets cancelled → client/server desync.

Fix:
- In the QUICK_MOVE guard, **drop the `.getItem().isEmpty()` test entirely.** A QUICK_MOVE on a genuinely empty slot is a harmless no-op the server ignores; it's not what causes the 2b2t kick.
- Cancel a QUICK_MOVE only when the slot index is actually invalid: `slot < 0 || slot >= menu.slots.size()`, or when `menu == null`.
- Strengthen the genuine stale-click guard instead: cancel a click whose `containerId()` doesn't match the currently open menu's container id (a click for a container that's no longer open) — that's the real source of the inventory kick. (If `AbstractContainerMenu.containerId` isn't directly accessible in Mojang mappings, add a small accessor mixin or compare via the open `AbstractContainerScreen`'s menu. ⚠ verify accessibility.)
- Keep `block-no-screen-clicks` and the per-tick cap as-is.
- If `ElytraReplace` is deleted (section 6), remove its name from the doc-comment's list of inventory modules.

## 5. Completely redo `VerticalVelocityTracker` — track only, never alter

The current module modifies movement. The new one must be **purely passive**: it observes and displays vertical velocity and acceleration and never touches movement. Rewrite from scratch:

- No `PlayerMoveEvent` handler that sets movement, no `setDeltaMovement`, no `IVec3d` writes — nothing that changes the player's motion.
- On `TickEvent.Post`, compute **actual** vertical velocity from the position delta (more accurate than `getDeltaMovement` for real movement): `vY = (mc.player.getY() - prevY) * 20.0` (blocks/sec). Store `prevY` each tick.
- Acceleration: `aY = (vY - prevVY)` per tick × 20, i.e. blocks/sec². Store `prevVY`.
- Optional light smoothing (e.g. an exponential moving average with a `smoothing` setting) — but the raw values must remain available.
- Display: override `getInfoString()` to show current `vY` (and optionally `aY`), and add a Meteor `HudElement`/HUD entry showing both, formatted (e.g. `vY: +12.3 b/s  aY: -1.4 b/s²`). Use whatever HUD-element pattern the rest of the addon already uses.
- Reset `prevY`/`prevVY` on `onActivate()` and when the player is null.
